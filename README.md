# KeyForge — Developer API Key & Access Management Platform

A multi-tenant portal where organizations create projects, generate scoped API keys, monitor
usage, and revoke/rotate keys — the internal-platform-team equivalent of "Stripe Dashboard >
Developers > API Keys."

This is a **Round 2 hackathon challenge codebase**. Roughly 70% of the app is fully working end to
end; the rest is intentionally buggy or incomplete so you have real problems to solve in a fixed
window. Nothing here is pseudocode — everything compiles and runs once dependencies are installed
and a database is available.

---

## 1. Project Overview

KeyForge lets an organization:

- Create **Organizations** with plan tiers (FREE / PRO / ENTERPRISE) and invite teammates with
  org-scoped roles (OWNER / ADMIN / MEMBER).
- Create **Projects** under an organization, each tagged with an environment
  (DEVELOPMENT / STAGING / PRODUCTION).
- Generate **API Keys** scoped to specific permissions (`READ_USERS`, `WRITE_BILLING`, etc.), with
  a configurable per-minute rate limit and optional expiry.
- See the full key **exactly once** at creation time, then only ever see a masked prefix
  afterward — the platform stores a SHA-256 hash, never the plaintext.
- **Revoke** keys instantly (soft delete via status change).
- Watch **usage analytics** (calls per day, error rates) and raw usage logs, generated by a real
  demo endpoint (`/api/demo/protected-resource`) that requires a valid key to call.

## 2. Architecture

### Frontend
Angular 20, standalone components (no NgModules), Angular Material, Angular Signals for
session/org state and the dashboard's polling live counter, RxJS for HTTP + reactive streams.
Routes are lazy-loaded via `loadComponent`. A functional `authInterceptor` attaches the JWT to
every request and bounces to `/login` on 401. Functional guards (`CanActivateFn`) protect
authenticated routes and org-role-restricted routes.

### Backend
Java 21, Spring Boot 3.3, layered architecture: `Entity → Repository (Spring Data JPA) →
Service → Controller → DTO`. Spring Security with a stateless JWT filter chain
(`JwtAuthenticationFilter`) handles user auth; a second, independent filter (`ApiKeyAuthFilter`)
guards the simulated protected resource using the platform's own generated API keys. Bean
Validation on request DTOs. A single `@RestControllerAdvice` (`GlobalExceptionHandler`) normalizes
every error response to one JSON shape.

### Database
PostgreSQL. See `docs/DATABASE-SCHEMA.sql` for the canonical DDL and `docs/ER-DIAGRAM.md` for the
entity-relationship diagram. Hibernate runs with `ddl-auto=update` for local dev convenience, but
the SQL file is the source of truth for CI/from-scratch setups.

### API Flow
1. `POST /api/auth/register` or `/login` → JWT returned, stored client-side, attached to every
   subsequent request via `Authorization: Bearer <token>`.
2. Every org-scoped endpoint (projects, keys, usage) re-checks the caller's `OrganizationMembership`
   server-side in the service layer via `OrganizationAccessService` — the frontend guard is a UX
   convenience, not the source of truth.
3. `POST .../keys` generates a key, returns the plaintext exactly once, and persists only its
   SHA-256 hash + a short visible prefix.
4. Calling `GET /api/demo/protected-resource` with `X-API-Key: <full key>` goes through
   `ApiKeyAuthFilter`, which resolves the key, records a `ApiKeyUsageLog` row, and updates the
   key's rate-limit window counter.
5. The dashboard, key list, and analytics pages read that data back out through
   `UsageAnalyticsService` / `ApiKeyUsageLogRepository`.

### Folder Structure
See `docs/FOLDER-STRUCTURE.md` for the full annotated tree of both `backend/` and `frontend/`.

## 3. Existing Features (already working)

**Backend**
- JWT auth: register + login, end to end.
- Spring Security config with org-membership-level role checks (`OrganizationAccessService`).
- Full CRUD for Organization, Project, ApiKey through the standard layered architecture.
- API key generation: secure random key, full key shown once, SHA-256 hash stored thereafter.
- Bean Validation on DTOs.
- `@RestControllerAdvice` global exception handler with a consistent error JSON shape.
- Pagination (`Pageable`) on the API keys list and usage logs list endpoints.
- Key revocation endpoint (status → `REVOKED`, soft delete).
- Usage logging filter that records `ApiKeyUsageLog` rows on every call to the demo protected
  resource.
- Usage analytics endpoint (calls per day for a key).
- Startup data seeder (`DataSeeder`, a `CommandLineRunner`) plus a standalone `seed-data/seed.sql`.

**Frontend**
- Login/Register pages, wired to the backend, JWT persisted client-side.
- Functional route guards for auth and org-role gating.
- Main layout: `mat-sidenav` + `mat-toolbar`, organization switcher in the top bar.
- Dashboard: stat cards (calls today, active keys, error rate), signal + `interval()` polling.
- Projects list (Material table).
- API Keys list (per project): Material table, pagination, status chips.
- Create API Key dialog: name, scope checkboxes, expiry date picker, rate limit input.
- Key-created reveal screen: shows the full key once, copy-to-clipboard, "you won't see this
  again" warning.
- Usage analytics page: summary cards + daily breakdown table.
- `AuthService`, `OrganizationService`, `ProjectService`, `ApiKeyService`, `UsageService` — typed
  HttpClient wrappers matching the backend DTOs.
- Signals used for: current user/org (`SessionStateService`), dashboard's live-ish usage counter.
- JWT auth interceptor.

## 5. Missing Features (to implement)

Marked with `TODO` in the code, or a controller method that returns `501 Not Implemented` /
throws `UnsupportedOperationException`:

- **Backend — key rotation.** `ApiKeyService.rotateApiKey()` is a stub; the controller endpoint
  exists but currently just returns `501`. A real implementation needs to generate a new key, keep
  the old one valid for a grace period, then auto-expire it.
- **Backend — rate limit enforcement.** The counting/tracking exists (see Known Bug #4) but
  nothing currently rejects a request once `rateLimitPerMinute` is exceeded. `ApiKeyAuthFilter`
  has a comment marking exactly where a 429 should be returned.
- **Backend — platform admin cross-org analytics.** `AdminController.platformUsageSummary()` is a
  stub (`501`). No service/repository support exists yet for aggregating usage across every
  organization on the platform.
- **Frontend — key rotation UI.** The API Keys list has a rotate button that just shows a "coming
  soon" snackbar. There's also an unrouted draft component
  (`features/key-rotation/key-rotation-panel.component.ts`) that got partway there before the
  backend endpoint was confirmed to be unimplemented — you can start from it or from scratch.
- **Frontend — audit log page.** `features/audit-log/audit-log.component.ts` is a placeholder
  shell with no data fetching. The backend already writes `AuditLog` rows on key create/revoke
  (see `AuditLogRepository`), but there's no read endpoint yet — you'll likely need to add one.
- **Frontend — scope-checkbox reactivity.** Tied to Known Bug #6: fixing this is both a bug fix
  and part of finishing the create-key form properly (e.g. migrating to a real `FormArray` so
  checkbox state and submitted payload can't drift apart).
- **Security — webhook signing.** Not implemented anywhere. If KeyForge ever notifies external
  systems (e.g. "key about to expire", "key revoked"), those webhook payloads should be signed
  (HMAC) so receivers can verify authenticity. Purely a documented enhancement opportunity — no
  scaffolding exists for it.

## 6. Enhancement Opportunities

Not required, but fair game if you have time left:

- Replace the dashboard's polling `interval()` with a WebSocket or Server-Sent Events channel for
  genuinely live usage counters.
- Turn `Scope` from an enum into a first-class entity if you want organizations to define custom
  permission scopes instead of the fixed built-in set.
- Add per-plan-tier limits (e.g. FREE orgs capped at N projects or N active keys).
- Add key rotation grace-period configuration per plan tier.
- Webhook signing (HMAC) for outbound notifications, as above.
- Structured audit log filtering/search once the read API exists.
- Response caching for the dashboard stats endpoint.
- Real chart library (e.g. a lightweight canvas chart) instead of/alongside the analytics table.

## 8. Setup Instructions

### Prerequisites
- Java 21 (JDK)
- Maven 3.9+
- Node.js 20+ and npm 10+
- PostgreSQL 15+ (local install or Docker)
- Angular CLI 20 (`npm install -g @angular/cli`) — optional, `npx` works too

### Database
```bash
createdb keyforge
# Apply schema (or just let Hibernate ddl-auto=update create it on first boot):
psql -d keyforge -f docs/DATABASE-SCHEMA.sql

# Optional: load realistic seed data (orgs, projects, keys, usage logs)
psql -d keyforge -f seed-data/seed.sql
```

If you'd rather start from an empty DB and let the app seed itself, skip `seed.sql` — the backend's
`DataSeeder` (`CommandLineRunner`) will populate demo data automatically on first boot **only if
the `users` table is empty**. Don't run both against the same database at once (duplicate/
conflicting IDs).

### Environment Variables (Backend)
Copy `backend/.env.example` to `backend/.env` (or export the vars directly / configure them in
your run configuration — Spring Boot reads these via `application.yml`'s `${VAR:default}` syntax,
so a plain `.env` file isn't auto-loaded by Spring itself; use your shell, an IDE run config, or a
tool like `direnv`/`dotenv-cli` to export them before running):

| Variable | Description | Example |
|---|---|---|
| `DB_URL` | JDBC URL | `jdbc:postgresql://localhost:5432/keyforge` |
| `DB_USERNAME` | DB user | `keyforge_user` |
| `DB_PASSWORD` | DB password | `keyforge_pass` |
| `JWT_SECRET` | HMAC signing secret, 256-bit+ | (generate a random string) |
| `JWT_EXPIRATION_MS` | Token lifetime in ms | `86400000` (24h) |
| `SERVER_PORT` | Backend port | `8080` |
| `CORS_ALLOWED_ORIGINS` | Comma-separated allowed origins | `http://localhost:4200` |

### Backend
```bash
cd backend
cp .env.example .env   # then export its values, see note above
mvn clean install
mvn spring-boot:run
# API now listening on http://localhost:8080
```

### Frontend
```bash
cd frontend
npm install
npm start   # ng serve, http://localhost:4200
```
The frontend's `src/environments/environment.ts` points at `http://localhost:8080/api` by default.

### Running Locally (full stack)
1. Start PostgreSQL, create the `keyforge` database.
2. Apply `docs/DATABASE-SCHEMA.sql` (or let Hibernate do it).
3. Optionally load `seed-data/seed.sql` for realistic demo data.
4. `cd backend && mvn spring-boot:run`
5. `cd frontend && npm install && npm start`
6. Visit `http://localhost:4200`, register a new account (or log in with a seeded user —
   password `Password123!` for all seeded accounts, see `seed-data/seed.sql`).
